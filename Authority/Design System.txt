## 1. Production-harden the code

The prototypes were built for visual impact, not production readiness. Clean up the technical debt while keeping the design identical:

**Images:**
- Replace all `<img>` tags for content images with Astro's `<Image>` component (from `astro:assets`). Import local images from `src/assets/images/`, use `<Image>` with explicit `width`, `height`, and `quality={80}`.
- Move any content images from `public/images/` to `src/assets/images/` and update imports.
- The hero image (above the fold) must have `loading="eager"`, `fetchpriority="high"`, and `decoding="sync"`. All other images use `loading="lazy"`.
- For any image that spans responsive breakpoints, add `widths` (1x and 2x array) and `sizes` props.

**Performance:**
- If any `<style>` blocks use `@apply` with brand tokens but are missing `@reference`, add it: `@reference "../../styles/global.css";` (adjust relative path as needed).
- Check for `will-change` in CSS — if it's applied broadly to many elements at load (e.g. on all `[data-reveal]` elements), remove it from the base CSS. It should only be added temporarily via JS right before animation.
- Add `content-visibility: auto` with `contain-intrinsic-size: auto 600px` to heavy below-fold sections (not the hero, not sticky elements, not anchor targets).
- Verify fonts are self-hosted via `@fontsource` packages in `global.css` — no Google Fonts `<link>` tags in the Layout or any component.

**Tokens:**
- Any hard-coded hex values, font names, or one-off color values in components should be moved to the `@theme` block in `global.css` as named tokens, then referenced via Tailwind classes.

Prompt B: Documentation and Inventory

Now the powerful part. This prompt creates three things that change how Claude works on your project going forward.

The project is cleaned up and the homepage is refactored into components. Now create the documentation and inventory. Do everything below in one go:

---

## 1. Create `docs/brand-guide.md`

Create a `docs/` folder in the project root. Inside it, create `brand-guide.md`.

Build this entirely from the actual homepage code, the `@theme` block, and my feedback from earlier in this session — not from any earlier brief files.

Include these sections:
- **Brand overview:** What this brand is, who it's for, the personality and mood
- **Color tokens:** Every color in the `@theme` block — token name, hex value, and when to use it
- **Typography:** Font families, size scale (in rem), weights, line heights, heading vs body usage rules
- **Spacing and layout:** Section padding, max content width, spacing scale, breakpoints
- **Visual tone:** 3-5 bullet points describing how the design should feel
- **Do's and don'ts:** Specific rules captured from my feedback
- **Image treatment:** How images should be styled to match the brand — border radius, shadows, overlays, filters, aspect ratios, and any effects used on the homepage. Describe the visual treatment, not where images are stored (that's handled separately).

Extract all values directly from the homepage code and `@theme` block — do not invent anything. This document should be specific enough that anyone reading it (human or AI) can make consistent design decisions.

---

## 2. Create `CLAUDE.md` in the project root

This file is read automatically by Claude Code in every session. Keep it lean — project context and workflow rules only. No hex values, no font specs — those live in the brand guide.

Include:
- **Project overview:** One paragraph — what this site is, who it's for, the tech stack (Astro 6 + Tailwind + React, deployed on Cloudflare Workers)
- **Documentation:** All brand, color, and typography decisions are in `docs/brand-guide.md` — read it before making any design decisions
- **File structure:**
  - Components: `src/components/` — PascalCase `.astro` files (e.g. `HeroSection.astro`)
  - Pages: `src/pages/` — lowercase with hyphens (e.g. `about.astro`)
  - Styles and tokens: `src/styles/global.css` under the `@theme` block
- **Component rules — IMPORTANT:**
  - Before building any new UI element, check `/design-system` for an existing component
  - If a match exists: import and reuse it — never rebuild what already exists
  - If no match exists: build the component in `src/components/`, add it to the design system page with its name and file path, THEN use it in the page you're building
  - The design system page must always reflect every reusable component in `src/components/` — keep it updated
- **Styling rules:**
  - All colors and fonts are defined as Tailwind tokens in `src/styles/global.css` under `@theme`
  - Always use Tailwind utility classes with those tokens — never hard-code hex values or font names in components
  - To add a new color or font, add it to `@theme` first, then use the class
- **Image handling:**
  - Content images (photos, hero images, team photos, blog thumbnails) belong in `src/assets/images/` — NOT in `public/`. If images are found in `public/images/`, move them to `src/assets/images/` and update all references.
  - Always use Astro's `<Image>` component (from `astro:assets`) for content images stored in `src/assets/`. Import the image file, then pass it to `<Image>` with explicit `width` and `height` props matching the intended display size. This gives automatic format conversion, responsive srcset, and lazy loading.
  - For external images from R2 buckets: first add the R2 domain to `image.domains` or `image.remotePatterns` in `astro.config.mjs`, then use Astro's `<Image>` component the same way as local images. Never use `<img>` as a shortcut just because the config is missing — update the config instead.
  - For third-party URLs you don't control (not R2): use a standard `<img>` tag with explicit `width` and `height`.
  - Always include these props on every `<Image>`: `width`, `height`, `quality={80}` (prevents blurry default compression). For any image that changes size across breakpoints or spans a responsive container, always include both `widths` (array of pixel widths for srcset — include 1x and 2x of the display size) and `sizes` (media query string matching your CSS breakpoints) — do not omit them on responsive images.
  - Above-the-fold images (hero, header) must use `loading="eager"`, `fetchpriority="high"`, and `decoding="sync"`. All other images use `loading="lazy"` (Astro default).
  - `public/` is only for system files that must be served at a fixed path: `favicon.ico`, `logo.svg`, `robots.txt`, OG images. No content images.
  - Always include descriptive `alt` text on every image.
- **Performance rules:**
  - **Fonts:** Must be self-hosted via `@fontsource` or `@fontsource-variable` packages imported in `src/styles/global.css`. Never add Google Fonts `<link>` tags or `preconnect` hints to external font services — these add 1-2 seconds to first paint via render-blocking cross-origin requests.
  - **`will-change`:** Avoid `will-change` entirely unless you are fixing a measured animation performance problem. Never put it in base CSS, never apply it to a selector that matches many elements, and never leave it on after an animation ends.
  - **Below-fold sections:** Use `content-visibility: auto` with `contain-intrinsic-size: auto 600px` only on heavy, self-contained sections that start below the fold. Do not apply it to headers, heroes, sticky elements, footer, anchor-link target sections, or sections whose JS needs layout measurements before they enter the viewport.
  - **No render-blocking external resources:** Do not add `<link rel="stylesheet">` to external domains in the `<head>`. All CSS and fonts must be same-origin (bundled into the `_astro/` directory).
- **Visual work:** When building or modifying any page, component, or visual element, load the `/frontend-design` skill first. This ensures distinctive, high-quality design — not generic AI output.
- **After every set of code changes:** run `npm run lint:fix` to auto-fix, then `npm run lint` to confirm no errors — do this before reporting done

## 3. Create the design system inventory at `src/pages/design-system.astro`

Build a no-index page that is an inventory of everything available to build with. This is the menu — anyone opening this page can see exactly what components, colors, and type styles exist and what they're called.

The page must:
- Include `<meta name="robots" content="noindex, nofollow">` so it doesn't appear in search results
- Have a clear heading: "Design System" with subtitle "Component inventory and brand reference"

**Sections to include:**

1. **Colors** — A swatch for every color token in `@theme`. Show the color visually, its token name (e.g. `walnut`), its Tailwind class (e.g. `bg-walnut`), and its hex value.
2. **Typography** — Render each heading level (h1–h4), body text, and small/caption text using the actual brand fonts and sizes from `@theme`.
3. **Components** — Import and render every reusable component from `src/components/`. Skip any internal or layout-only files that aren't meant to be used directly on pages. Group the rest in two sections:
   - **Atoms** — small reusable elements (buttons, badges, cards, tags, etc.)
   - **Sections** — full page sections (Hero, Features, CTA, Footer, Nav, etc.)

   For components that need content props to render, pass representative placeholder content. Under each rendered component, show its name and file path (e.g. `CTABanner` · `src/components/CTABanner.astro`).

Every component is imported from `src/components/` — nothing is defined inline on this page. This page is the source of truth for what exists and what it's called.

After creating all files, run `npm run lint:fix`, `npm run lint`, and `npm run build` to confirm everything passes.
